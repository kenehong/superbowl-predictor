# User Research Skill

## Core Rules
- **유저는 내가 아니다** — 내 사용 패턴을 유저에게 투영하지 않음
- **Verify with evidence** — "유저가 원할 것이다"는 가정. 데이터/사례로 검증
- **행동 > 의견** — 유저가 "뭘 원한다"보다 "실제로 뭘 하는가"가 중요

## Confidence Levels
| Level | 기준 | 행동 |
|-------|------|------|
| 🟢 HIGH | 실제 유저 데이터, 경쟁사 리뷰 분석, 관찰 기반 | 페르소나/저니맵에 반영 |
| 🟡 MEDIUM | 유사 서비스 패턴, 일반 UX 원칙 | 가설로 포함, "검증 필요" 표기 |
| 🔴 LOW | 추측, 개인 경험 일반화 | 가설 목록에만 포함. 의사결정에 사용 금지 |

## Protocol

### Phase 1: Hypothesis — 가설 설정
**Good Questions:**
- 이 프로젝트가 해결하는 문제는 무엇인가?
- 이 문제를 겪는 사람은 누구인가? (구체적으로)
- 현재 이 문제를 어떻게 해결하고 있는가?
- 우리 솔루션이 기존 방법보다 나은 이유는?
- 유저가 이걸 안 쓸 이유는? (anti-persona)

**Quality Gate:** 핵심 가설 3개 이내, 각각 검증 방법 정의

### Phase 2: Define — 타겟 유저 정의
**페르소나 프레임워크:**
```markdown
## 페르소나: [이름]
- **한줄 요약**: [나이대, 상황, 핵심 니즈]
- **목표**: 이 도구로 무엇을 달성하려는가?
- **맥락**: 어디서, 언제, 어떤 상황에서 사용하는가?
- **현재 방법**: 지금은 이 문제를 어떻게 해결?
- **좌절점**: 현재 방법의 불편한 점?
- **성공 기준**: "이렇게 되면 만족" (유저 관점)
```

**Anti-Persona:** 이 도구를 안 쓸 사람도 정의 (스코프 방지)

### Phase 3: Compete — 경쟁사 분석
**WebSearch로 유사 서비스 3~5개 조사:**
| 항목 | 조사 내용 |
|------|----------|
| 기능 | 핵심 기능, 차별 기능 |
| UX | 온보딩, 메인 플로우, 마찰점 |
| 가격 | 무료/유료, 제한 |
| 리뷰 | 유저가 좋아하는 것, 불만 |
| 약점 | 우리가 더 잘할 수 있는 것 |

**Quality Gate:**
- [ ] 경쟁사 3개 이상 분석
- [ ] 차별화 포인트 1개 이상 도출

### Phase 4: Map — 저니 맵
**유저 여정 단계:**
```
인지 → 진입 → 첫 사용 → 핵심 가치 체험 → 반복 사용 → 추천
```

**각 단계별:**
| 단계 | 유저 행동 | 유저 감정 | 터치포인트 | 개선 기회 |
|------|----------|----------|-----------|----------|
| 인지 | [어떻게 알게 되는가] | [기대/의심] | [채널] | |
| 진입 | [첫 화면에서 뭘 하는가] | [혼란/명확] | [UI] | |
| ... | | | | |

### Phase 5: Insight — 인사이트 도출
**Output Format:**
```markdown
## 유저 리서치 결과

### 타겟 유저
[페르소나 요약 — 1줄]

### 핵심 인사이트
1. [발견] — (Confidence: 🟢/🟡/🔴) [근거]
2. [발견] — (Confidence) [근거]
3. [발견] — (Confidence) [근거]

### 경쟁사 분석
| 서비스 | 강점 | 약점 | 우리 기회 |
|--------|------|------|----------|

### 유저 저니 맵
[단계별 표]

### 검증 필요 가설
- [가설]: [검증 방법]

### 디자인 임플리케이션
- [인사이트 → 구체적 UI/UX 결정]
```

**Quality Gate:**
- [ ] 페르소나 1줄 요약 존재
- [ ] 경쟁사 3개+ 분석 완료
- [ ] 인사이트에 Confidence level 표기
- [ ] 🔴 LOW 인사이트가 핵심 결정에 사용되지 않음
