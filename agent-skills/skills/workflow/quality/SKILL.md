# Quality Skill

## Core Rules
- **유저 시점 테스트** — 개발자가 아닌 첫 방문 유저 관점으로 테스트
- **Execute only when verified** — 버그 수정 후 반드시 재테스트. 추측으로 "고쳤다" 선언 금지
- **엣지 케이스 필수** — Happy path만 테스트하면 QA가 아님
- **눈으로 확인** — UI 변경 시 스크린샷 캡처로 시각적 검증 필수

## Confidence Levels
| Level | 기준 | 행동 |
|-------|------|------|
| 🟢 HIGH | 직접 테스트하여 동작 확인, 재현 가능 | "확인됨" 보고 |
| 🟡 MEDIUM | 코드 리뷰로 논리적으로 판단 | "코드상 정상이나 실제 테스트 권장" |
| 🔴 LOW | 추측, 유사 케이스에서 유추 | "미확인 — 반드시 수동 테스트 필요" |

## Protocol

### Phase 1: Plan — 테스트 계획
**Good Questions:**
- 타겟 유저는 누구인가?
- 유저의 메인 플로우(진입→결과)는 무엇인가?
- 가장 자주 사용하는 기능 Top 3는?
- 어떤 입력이 예상 밖으로 들어올 수 있는가?
- 네트워크 없는 환경에서 동작해야 하는가?

**Action:** 시나리오 3~5개 작성
**Quality Gate:** 시나리오가 Happy path + 에러 path 모두 커버

### Phase 2: Flow Test — 메인 플로우 테스트
**첫 방문 유저 시뮬레이션:**
1. 페이지 첫 로드 — 빈 상태가 의미 있는가? 안내가 충분한가?
2. 첫 입력 — 무엇을 어디에 입력해야 하는지 명확한가?
3. 결과 확인 — 결과가 기대와 일치하는가?
4. 재방문 — 이전 상태/히스토리가 보존되는가?

**Quality Gate:**
- [ ] 첫 방문자가 안내 없이 사용 가능
- [ ] 로딩 상태가 표시됨
- [ ] 결과가 정확하고 가독성 좋음
- [ ] 히스토리/상태 복원 정상
- [ ] Spacing — 섹션 간 1.25rem+, 카드 내부 0.75rem+, 스택 0.5rem+, 붙어있는 곳 없음

### Phase 3: Edge Cases — 엣지 케이스
**필수 테스트 목록:**
| 카테고리 | 테스트 항목 |
|---------|-----------|
| 빈 입력 | 아무것도 입력 안 하고 실행 |
| 잘못된 입력 | 형식에 맞지 않는 데이터 |
| 긴 입력 | 비정상적으로 긴 텍스트/숫자 |
| 특수문자 | `<script>`, `"`, `'`, `&` 등 XSS 시도 |
| 네트워크 실패 | API 호출 실패 시 에러 처리 |
| localStorage 가득 | 저장 공간 부족 시 |
| 모바일 | 터치, 작은 화면, 가로/세로 |
| Spacing | 섹션/카드/스택 요소 간 여백 충분한지, 빽빽하게 붙은 곳 없는지 |
| 중복 실행 | 버튼 연타, 빠른 연속 요청 |
| 비주얼 회귀 | 스크린샷 캡처 → 이전 버전과 시각 비교 |

### Phase 4: Visual Verify — 스크린샷 검증
**UI가 포함된 프로젝트에서 필수 (→ screenshot.md 참조)**

**자동 트리거 조건:**
- 유저가 "이상해", "깨졌어", "안 맞아", "looks wrong", "spacing is off" 등 언급
- UI/XAML/CSS 변경 후 결과 확인 필요
- Figma 디자인과 구현 비교

**워크플로우:**
1. **캡처** — 앱 윈도우 스크린샷 촬영
   - Windows: `python screenshot.py "<window_title>" output.png`
   - Mac: `screencapture -l <windowID> output.png`
2. **비교** — 캡처된 이미지를 Figma 디자인 또는 기대 레이아웃과 대조
3. **체크리스트:**
   - [ ] Spacing이 디자인과 일치하는가?
   - [ ] 정렬(alignment)이 정확한가?
   - [ ] 컬러/폰트가 스펙과 일치하는가?
   - [ ] 빈 상태/에러 상태 UI가 적절한가?
   - [ ] 화면 크기 변경 시 레이아웃 깨지지 않는가?
4. **수정 후 재캡처** — 변경 → 스크린샷 → 비교를 PASS까지 반복

**멀티페이지 리뷰 (GIF 녹화):**
- 여러 화면/인터랙션을 한번에 검증할 때 사용
- `python screen_record.py "<title>" review.gif --duration 10 --fps 2`
- 네비게이션 흐름, 페이지 전환, 애니메이션 확인

### Phase 5: Fix — 버그 수정
- 발견 즉시 수정
- 수정 후 해당 시나리오 **재테스트** (🟢 HIGH 확인까지)
- UI 수정 시 → Phase 4 스크린샷 재캡처로 시각적 확인
- 수정이 다른 기능을 깨뜨리지 않았는지 확인

### Phase 6: Report — QA 보고
**Output Format:**
```markdown
## QA 결과

### 테스트 환경
- 브라우저/환경: [...]
- 테스트 일시: [...]

### 시나리오 결과
| # | 시나리오 | 결과 | Confidence |
|---|---------|------|-----------|
| 1 | [설명] | PASS/FAIL | 🟢/🟡/🔴 |

### 발견된 버그
| # | 설명 | 심각도 | 수정 여부 |
|---|------|-------|----------|

### 미확인 항목 (수동 테스트 필요)
- [항목]: [이유]
```

**Quality Gate:**
- [ ] 모든 시나리오 PASS
- [ ] FAIL 항목 0개 (또는 수정 완료)
- [ ] 🔴 LOW confidence 항목 별도 표기
