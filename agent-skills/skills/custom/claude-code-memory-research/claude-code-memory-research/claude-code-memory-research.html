핵심 요약 (Executive Summary)

Claude Code의 메모리 시스템은 "텍스트 주입"이다. 매 세션마다 관련 마크다운 파일이 프롬프트 컨텍스트에 로드된다. 이것은 "기억"이 아니라 "매번 다시 읽는 지시서"이다. 이 근본적 이해가 최적화의 출발점이다.

1. CLAUDE.md는 150줄 이하로

300줄 비구조화 파일은 4,500 토큰, 최적화하면 1,800 토큰 (60% 절감). 커뮤니티 합의: 프로젝트 CLAUDE.md는 150줄, 글로벌은 50줄 이하.

2. 모듈형 규칙으로 분리

.claude/rules/*.md에 주제별 파일 분리. glob 패턴으로 조건부 로딩하면 컨텍스트 노이즈 30~40% 감소.

3. Skills는 온디맨드 로딩

스킬은 시작 시 이름/설명만 로드. 호출 시에만 전체 SKILL.md를 주입. 전문 지시를 스킬로 이동하면 세션당 15,000 토큰 절약 가능.

4. Auto Memory 활용

MEMORY.md 첫 200줄만 로드됨. 상세 내용은 토픽 파일로 분리, 필요 시에만 Claude가 읽음. "remember that..."으로 직접 저장 지시 가능.

리스트 > 산문

Claude Code는 서술형 문단보다 불릿 리스트를 훨씬 효율적으로 파싱함.

금지 사항 명시

"X를 하지 마라" 형태의 명시적 금지가 긍정형 권고보다 에러 방지에 더 효과적.

코드 예시 3~5개

원하는 출력 포맷의 코드 예시를 보여주면 설명 대비 수정 요청이 40% 감소.

권장 CLAUDE.md 스켈레톤

# 프로젝트명 ## Stack - Next.js 15, TypeScript 5.6, Tailwind v4 - DB: PostgreSQL + Drizzle ORM - Test: Vitest + Playwright ## Commands - Build: `pnpm build` - Test: `pnpm test` - Lint: `pnpm lint --fix` ## Key Directories - src/app/ - App Router pages - src/lib/ - Shared utilities ## Code Style - Functional components only - 2-space indent, max 100 chars/line - NEVER use `any` type

.claude/rules/ 모듈형 규칙

.claude/ ├── CLAUDE.md # 프로젝트 개요 └── rules/ ├── code-style.md # 포맷팅, 네이밍 ├── testing.md # 테스트 요구사항 └── security.md # 보안 가이드라인

조건부 로딩 (Glob 패턴)

--- paths: - "**/*.test.ts" --- # Testing Rules - describe / it 사용 (test 아님) - MSW로 네트워크 모킹

효과: 조건부 규칙 사용 시 시스템 프롬프트 노이즈 30% 감소, 지시 적중률 35% 향상.

논쟁점

CLAUDE.md 길이 제한에 대한 논쟁

찬성: 토큰 절약, 지시 누락 방지. 반론: 복잡한 프로젝트는 150줄로 부족 → .claude/rules/로 해결.

결론: 합의 코어는 짧게, 세부는 rules/로 분리. 전체 합산 10,000 토큰 이내.

Skills 시스템 완전 해부

스킬이란 무엇인가

스킬은 "프롬프트 확장 + 컨텍스트 수정"이다. 호출 시에만 대화 컨텍스트에 주입되는 지시 템플릿.

핵심: Tool = 즉각 실행 (~100 토큰), Skill = 워크플로우 가이드 (~1,500+ 토큰). 스킬은 name/description만 상시 로드.

SKILL.md 구조

--- name: explain-code description: 코드를 시각적 다이어그램과 비유로 설명 --- # 코드 설명 스킬 1. 비유로 시작 2. 다이어그램 생성 3. 코드 워크스루 4. 주의점 강조

YAML Frontmatter 필드

필드필수설명
nameO/slash-command 이름
descriptionO스킬 선택 판단 기준 (가장 중요)
allowed-toolsX허용 도구 범위
modelX모델 오버라이드

내부 동작 방식

  1. 시작 시: name + description만 수집 → <available_skills> XML에 포함
  2. 호출 시: Claude가 Skill 도구를 호출
  3. 주입: SKILL.md body + base path를 tool_result로 반환
  4. 실행: 주입된 지시에 따라 작업 수행

스킬 선택: 알고리즘이 아닌 Claude의 의미적 매칭. description 작성이 트리거 정확도의 핵심.

토큰 최적화 전략

54% 토큰 절감 사례

John Lindquist의 구조 재편: 초기 컨텍스트 7,584 → 3,434 토큰 (54% 절감).

기법 1: 트리거 테이블 치환

BEFORE
When the user mentions tasks, tracking, or backlog management, you should use the beads system... (수백 줄)
AFTER
| Triggers | Skill | Tools | |----------|-------|-------| | task, track | beads | beads_ready | | debug, error | investigate | ... |

기법 2: Lazy Loading via Skill()

# 상세 프로토콜을 조건부로 참조 - 디버깅 워크플로우 -> Skill('investigate') - 태스크 관리 -> Skill('beads')

12가지 실전 토큰 절약 기법

1. 작업별 모델 선택

단순=Haiku, 복잡=Opus. /model로 수동 선택.

2. /clear로 초기화

새 작업 시작 시 이전 히스토리 정리.

3. 50%에서 /compact

/context로 모니터링, 50% 도달 시 요약.

4. .claudeignore 설정

node_modules/, build/ 등 불필요 디렉토리 제외.

5. 구체적 프롬프트

"src/login.ts 47번 줄 수정" > "어딘가 버그"

6. 파일 경로 직접 지정

"Read src/services/user.ts" → 검색 오버헤드 제거.

7. 서브 에이전트 활용

복잡한 작업을 독립 에이전트에 위임.

8. Headless 모드

claude --print "run tests" CI/CD용.

2. Memory Bank 패턴

가장 혁신적인 패턴. CLAUDE.md를 "살아있는 문서"로 활용:

4단계 지식 루프

매 세션 종료 시 /update-memory로 학습 내용을 구조화 파일에 기록.

논쟁점

복잡한 워크플로우 vs 바닐라

찬성: 대규모 프로젝트에서 일관성 보장. 반론: "vanilla cc is better than any workflows with smaller tasks." 작은 작업 분할이면 충분.

결론: 상황 의존적 솔로/소규모 = 바닐라 + 간단한 CLAUDE.md. 팀/엔터프라이즈 = 구조화 필요.
