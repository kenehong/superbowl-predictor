# Instructions for GitHub Copilot

## Tone and Style

- Write in a concise, executive-friendly tone
- Avoid jargon; explain technical concepts simply
- Use bullet points for key takeaways
- Keep paragraphs short (3-4 sentences max)

## Document Structure

- Start with a clear objective statement
- Include a "Problem Overview" section before proposing solutions
- End with concrete "Next Steps" or "Recommendations"

## Terminology

- Use "product manager" not "PM"
- Use "customer" not "user" when discussing external stakeholders
- Refer to our team as "<YOUR TEAM NAME>"

## Templates

- Model specs after existing documents like <PATH TO YOUR SPECS FOLDER>/PMSpecTemplate.md

GitHub Copilot automatically reads this file and applies these instructions to everything it writes.

## Skills

- Use `/write-pm-spec <topic>` to start an interactive PM spec writing workflow.
  This skill walks through discovery, research, planning, and drafting following
  the PMSpecTemplate.md format and the tone/terminology rules above.
